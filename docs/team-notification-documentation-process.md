## 📢 重要通知

**文档维护流程已正式建立！**

为了确保项目文档与实际实现保持一致，我们建立了完整的文档维护流程。请所有团队成员了解并遵循此流程。

---

## 🎯 为什么需要文档维护流程？

**问题：**
- 文档与实际实现不一致（如 Tiptap → MarkdownEditor 未及时更新）
- 技术栈变更未反映到文档中
- 架构模式变更未记录

**解决方案：**
- 建立清晰的文档维护流程
- 明确角色和职责
- 建立定期审查机制

---

## 📋 关键流程要点

### 1. 文档分类和责任

| 文档类型 | 维护责任人 | 更新触发条件 |
|---------|-----------|-------------|
| **PRD、Epic、Story** | **PM** | 功能需求、技术栈、架构变更 |
| **架构文档、技术规范** | **Architect** | 架构、技术实现变更 |
| **用户文档、测试指南** | **Tech Writer / UX** | 功能、界面、流程变更 |

### 2. 代码变更时的检查

**每次代码提交前，请检查：**

- [ ] 技术栈变更？→ 更新 PRD、相关 Epic
- [ ] 架构模式变更？→ 更新架构文档、相关 Epic
- [ ] 功能变更？→ 更新相关 Epic、Story
- [ ] API 变更？→ 更新 API 文档

**快速检查清单：** 参考 `docs/documentation-update-checklist.md`

### 3. 更新优先级

- **P0（必须立即更新）** - 技术栈、架构、核心功能变更
- **P1（建议尽快更新）** - 功能增强、次要功能变更
- **P2（定期批量更新）** - 文档完善、格式优化
